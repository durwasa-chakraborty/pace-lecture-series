# Prompt: PACE Lecture Series — Speaker Self-Service Portal

## Role

You are designing and building a speaker self-service portal for the **PACE Lab Research Huddle**
lecture series at IIT Madras. **Your first deliverable is a plan, not code.**

## Where things stand today

The repo at `~/programming/pace-lecture-series` is a Hugo site (Mainroad theme) deployed to AWS
Amplify via CDK. Publishing a talk today is a manual, single-operator flow:

| Piece | What it does |
| --- | --- |
| `next_talk.template` | Hand-edited key/value file: `DATE`, `SPEAKER`, `TITLE`, `VENUE`, `TIME`, `GMEET`, `ABOUT`, `ABSTRACT` |
| `publish_talk.ml` | OCaml script: parses the template, writes `content/talks/YYYY-Mmm-DD-talk-1.md`, runs `hugo`, then `git add`/`commit`/`push` |
| `next_talk_email.txt` | The announcement email, generated by the same script |
| `cdk/` | AWS CDK stacks (`cdk-stack.ts`, `hugo-stack.ts`) — Amplify hosting, CI/CD from `main` |

So the two automated steps are **"generate the talk page"** and **"generate the announcement
email."** Everything upstream of those — a speaker telling us they want a slot, picking a date,
booking a room, sending the calendar invite — is still done by hand, over email, by one person.

That is the gap to close.

## What to build

A web app where a prospective speaker picks an open date from a calendar and submits their talk.
An admin (me) reviews and approves it. Approval then drives the downstream automation, and everyone
can watch it happen.

### 1. Speaker-facing submission

- A calendar view showing which dates are **open**, **pending**, and **booked**. Open slots follow
  the existing cadence (weekday evenings, 5–6 pm IST); a booked date must not be selectable.
- Selecting an open date opens a form:
  - **Speaker name**
  - **Talk title**
  - **Abstract**
  - **About the speaker** — LinkedIn / blog / personal site URL
  - Contact email (needed to send them the invite and to reach them if the slot changes)
- On submit: the request lands in a `pending` state. No calendar event, no email, no site change yet.
  Nothing is published on the speaker's say-so alone.

### 2. Admin approval

- An authenticated admin view listing pending submissions with everything the speaker entered.
- The admin can **approve**, **reject** (with a reason back to the speaker), or **edit-then-approve**
  — abstracts arrive with typos, and I'd rather fix them than bounce the submission.
- Only an approval triggers the downstream pipeline. Treat that as the single trust boundary in the
  system.

### 3. The pipeline, once approved

Run these as discrete, individually-retryable stages — not one monolithic "do everything" call.
If stage 3 fails, stages 1 and 2 must not re-run.

1. **Publish the talk page** — produce `content/talks/YYYY-Mmm-DD-talk-1.md` in the exact format
   `publish_talk.ml` emits today, commit it (message must match `^(DEV|INFO|MEET|DOC): .+`, see
   `README.md`), and push. Amplify deploys from `main`.
2. **Create the Google Calendar event** — title, abstract, time, venue, Google Meet link, with an
   invite emailed to the speaker and the mailing list.
3. **Request the room booking** — email KC sir asking for the lecture hall, with the date, time, and
   expected headcount.
4. **Send the announcement email** — the `next_talk_email.txt` content, to the list.

### 4. Status UI

Both the admin and the speaker should see, per talk, which stages have completed:

```
Talk: "Making sense of eBPF" — Pragyansh Chaturvedi — Wed 15 Jul 2026

  [✓] Submitted            11 Jul 2026, 14:02
  [✓] Approved             11 Jul 2026, 18:31   (by durwasa)
  [✓] Site published       11 Jul 2026, 18:31   → /talks/2026-jul-15-talk-1/
  [✓] Calendar invite      11 Jul 2026, 18:32   → 4 accepted
  [⟳] Room booking         awaiting reply from KC sir
  [ ] Announcement email
```

A failed stage must say **what** failed and offer a retry — not just go red.

## Hard constraints

- **Testing mode.** Until I explicitly flip it, *every* outbound email and calendar invite goes to
  **`cs24d011@cse.iitm.ac.in`** and nowhere else. The real mailing list and KC sir's address must be
  configuration, not hardcoded strings — I want to change the recipients without changing code, and I
  want it to be *impossible* to accidentally mail the real list while in testing mode. Make that
  failure mode structurally unavailable, not merely unlikely.
- **Credentials.** This needs Google API access — Calendar (create events, invite attendees) and
  Gmail or an SMTP relay (send mail). Tell me exactly which OAuth scopes or service-account setup is
  required, whether domain-wide delegation is needed for the `cse.iitm.ac.in` workspace, and what I
  personally have to click through in Google Cloud Console. Assume I have not set any of this up.
- **Hosting.** AWS, consistent with the existing CDK setup. Secrets live in Secrets Manager or SSM
  Parameter Store — never in the repo, never in an env file that could be committed.
- **Don't break the current flow.** `publish_talk.ml` and the template files must keep working. The
  portal is a new front door to the same publishing path, not a replacement for it.

## Deliverable

Write a plan to **`~/programming/pace-speaker-portal/PLAN.md`** (create the directory; keep it out of
the lecture-series repo so the Hugo build isn't disturbed). Cover:

1. **Architecture** — the stack you'd pick and *why*, plus what you rejected. Weigh the operational
   cost honestly: this is a lab lecture series that runs maybe twice a month, and I am the only
   maintainer. A design I can't debug at 11pm is a bad design, however elegant.
2. **Data model** — talks, their states, the audit trail of who approved what and when.
3. **The approval → pipeline mechanism** — how stages are sequenced, made idempotent, and retried.
   Be specific about what happens when Google's API times out *after* creating the event.
4. **Auth** — how the admin logs in; whether speakers need accounts at all (I'd rather they didn't).
5. **Google API setup** — a literal checklist of what I click, in order.
6. **AWS infrastructure** — what CDK adds to the existing stacks.
7. **Build order** — milestones I can deploy and use one at a time. I want something working end to
   end early, even if it's ugly, over a beautiful system that lands all at once.
8. **Open questions and risks** — including anything below you think I've gotten wrong.
